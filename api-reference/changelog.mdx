Changes to the API will be documented below in reverse chronological order. Refer to the official release notes for the wider system updates (and API changes pre-2022).


<Update label="22nd December 2025" description="">

- Updated `DELETE /resale-listings` to properly cancel resale listings with validation. Response now includes `contractsCancelled`, `contractsSold`, and `contractsFrozen` counts. Sold and frozen contracts are preserved; only unsold, unfrozen contracts are cancelled.

</Update>

<Update label="19th December 2025" description="">

- Added `eventId` as a filter to the [GET /orders](/api-reference/orders/retrieve-orders) endpoint to retrieve only orders that contain items from the given event ID.
- Added a new `orderItemId` filter to [GET /v1/wallets/transactions](/api-reference/wallets/retrieve-wallet-transactions) endpoint.
- Added a new `source` attribute to the `WalletTransactions` entity from [GET /v1/wallets/transactions](/api-reference/wallets/retrieve-wallet-transactions) to detail the source of the wallet transaction.

</Update>

<Update label="17th December 2025" description="">

- Added new [GET /wallets](/api-reference/wallets/retrieve-wallets) and [GET /wallet](/api-reference/wallets/retrieve-a-wallet) endpoints to retrieve wallets registered within a company (NB: wallets power our account balance functionality).
- Added a new [GET /v1/wallets/transactions](/api-reference/wallets/retrieve-wallet-transactions) endpoint to list all recent transactions across wallets (or for a specific wallet)
- Added [POST /v1/wallets/transactions/confirm](/api-reference/wallets/confirm-a-wallet-transaction) to allow confirmation of pending transactions. Pending transactions are typically used for resale, order modifications, and other internal processes that deposit money into wallets.
- Added [POST /v1/wallets/payouts/initiate](/api-reference/wallets/initiate-a-payout) to allow initiation of an external payout from a wallet. Unlike cashouts, payouts are not capped at the mount of topups purchased. This endpoint allows for a full payout of the account balance, regardless of source of funds.
- Added [POST /v1/wallets/payouts/confirm](/api-reference/wallets/confirm-a-payout) to confirm when an initiated payout has been completed. This is an important step where external parties need to also verify that the payout was handled correctly.
- Removed deprecated endpoints `POST /v1/wallets/nfc-download` and `POST /v1/wallets/nfc-download/confirm` as these are no longer in use or supported.
- Add [GET /v1/resale/listings](/api-reference/resale/retrieve-resale-listings) index endpoint to return all available resale listings
- Payload updats to [POST /v1/resale/listings](/api-reference/resale/create-a-new-resale-listing) to support creation of multiple item listings
- Added a new `authenticationProviders` relationship that can be included on both [GET /customers](/api-reference/customers/retrieve-customers) and [GET /customer](/api-reference/customers/retrieve-customer) to determine the related customer SSO IDs.

</Update>

<Update label="3rd December 2025" description="">

- Added a new `orderColumn` attribute to [GET /sale-items](/api-reference/sale-items/retrieve-sale-items) endpoint, which reflects the sort order defined in admin, and used in the ticket shop.

</Update>

<Update label="20th November 2025" description="">

- Added a new `DELETE /customers` endpoint to support customer data deletion requests & accommodate data retention policies.

</Update>

<Update label="11th November 2025" description="">

- On [GET /events](/api-reference/events/retrieve-events) and [GET /event](/api-reference/events/retrieve-event) we now return the `isCancelled` attribute to indicate that an event has been marked as cancelled.

</Update>

<Update label="4th November 2025" description="">

- Added a new `DELETE /customers` endpoint to support customer data deletion requests & accommodate data retention policies.
- We now return all fees, resale fees, fulfilment fees, and protection fees via the [GET /v1/order-items](/api-reference/orders/retrieve-order-items) endpoint
- When returning fees via the [GET /v1/order-items](/api-reference/orders/retrieve-order-items) endpoint we also provide access to the `producingOrderItemId` field which allows consumers to reference the order item that generated that specific fee. Where this is `null` the fee applies to the whole basket.
- Added the ability to retrieve orders from the [GET /v1/order](/api-reference/orders/retrieve-order) endpoint using the reference, rather than the ID.

</Update>

<Update label="16th October 2025" description="">

- Introduced access control scanning / pairing restrictions to items that belong to an order which is currently being modified, or resold.
- [GET /v1/entry/tickets](/api-reference/access-control/retrieve-entry-tickets) receives a new status (`variable.status`) option `locked`, which indicates that the order is currently being modified. Items with this status should be considered _temporarily_ invalid for scanning. The status of these items could return to `sold` or `refunded` in the future.
- Added `POST /v1/resale-contracts` to create a resale contract, which allows an order item to be resold
- Added `GET /v1/resale-contract` to get a resale contract for an order item or by ID
- Added `POST /v1/resale-contracts/delete` to delete a resale contract for an order item or a resale contract ID
- Added `replacedOrderItem` to `GET v1/order-items`, `GET v1/order-item` and `GET v1/order` to return the original order item for resold and modified order items.

</Update>
