## Overview

Single Sign-On lets the people who work in your company's admin area sign in through your organisation's identity provider instead of (or alongside) an email and password. When SSO is enabled, an extra **Sign in with [provider]** button appears on the admin login screen, and staff authenticate with their existing work account rather than a separate platform password.

SSO applies to [admin users](/core-platform/system-access/users) — the staff who manage events, process orders, and run box office. It does **not** change how your customers sign in to the shop. Customer-facing sign-in is configured separately.

<Tip>New in Release 47</Tip>

SSO is useful when you want to:

- Centralise staff sign-in through your existing identity provider so access follows your organisation's accounts
- Remove separate admin passwords for some or all staff, and manage who can still use a password on a per-user basis
- Keep a record of how each admin user signed in

Currently, **Google Workspace** is the supported identity provider. The feature is built so that further providers can be added over time; where this page describes Google Workspace specifics (such as the workspace domain), other providers would expose their own equivalent settings when they become available.

## How It Works

SSO sits in front of the admin area's normal email-and-password login. The overall flow is:

```
1. Your account manager configures the SSO provider for your company
   └── Provider connection details + authorised redirect URI

2. Each admin user is given an authentication policy
   └── Password only · Password or SSO · SSO only

3. Staff visit the admin login screen
   └── A "Sign in with [provider]" button appears

4. Staff sign in with their identity provider account
   └── The platform matches them to an existing admin user and signs them in
```

Two important rules shape who can sign in:

- **Staff must already exist as an admin user.** SSO never creates new admin accounts. A person can only sign in with SSO if they have already been [added to your company](/core-platform/system-access/users) as a user with the same email address (or a previously linked provider account). If no matching admin user exists, sign-in is refused.
- **The identity provider account is matched to the admin user.** On a user's first SSO sign-in, their provider account is linked to their admin user. After that, the stored link is used to recognise them on future sign-ins.

<Note>
Signing in with SSO does not change which company is selected. Staff who have access to more than one company continue in the company they last had selected.
</Note>

## Configuring the SSO Provider

The provider connection is configured at the company level by your **account manager**, not from within your own admin area. If you want SSO enabled for your company, contact your account manager with your identity provider details.

When SSO is set up, the following are configured:

| Setting | What it controls |
|---------|------------------|
| **Enable admin Single Sign-On** | Turns SSO on for the company. While off, no SSO button appears and all staff sign in with a password. |
| **Integration type** | The identity provider to connect to. **Google Workspace** is currently available. |
| **Authorised redirect URI** | A read-only URL generated by the platform. It must be added to the authorised redirect URIs in the identity provider's OAuth client settings so the provider will return staff to the platform after sign-in. |
| **Google Workspace domain** | The workspace domain that sign-ins must belong to (for example `example.com`). Only accounts in this domain are accepted. |
| **Client ID** and **Client secret** | The OAuth credentials from the identity provider's application. |
| **Admin SSO identifier key** | Optional. A custom claim to use as the unique staff identifier. Left blank, the provider's user ID is used. |
| **Admin logout URL** | Optional. Where staff are sent after the provider's logout flow completes. |
| **Admin SSO name** | The label shown on the sign-in button, as **Sign in with [name]**. |

<Note>
The platform only trusts the identity provider's authoritative confirmation that an account belongs to the configured workspace domain and that the email address is verified. A self-declared email domain alone is not enough to sign in, which prevents external or unverified accounts from gaining admin access.
</Note>

<Warning>
A **Callback URL override** setting exists for development environments only, where some providers reject local domains. It must never be set in production — when it is, the platform shows a warning that it is for development purposes only.
</Warning>

### One Sign-In Button per Configuration

The admin login screen is shared, and several companies can use the same identity provider. To keep the screen simple:

- Companies that share the same SSO credentials are shown as a **single** sign-in option.
- Sign-in options are labelled with the **Admin SSO name** only. Company names are never shown on the login screen.

## Controlling Authentication per User

Once SSO is enabled for your company, you control how each admin user authenticates from their user detail page. This lets you move some staff to SSO while leaving others on a password — for example, rolling SSO out gradually, or keeping a password-based break-glass account.

The authentication controls appear only when SSO is enabled for the company, and only when editing **another** user — you cannot change your own authentication method.

To set a user's authentication method:

1. Open the user's detail page from [Users](/core-platform/system-access/users)
2. Click **Edit**
3. Find the **Authentication** section
4. Set the **Admin authentication policy**
5. Click **Submit**

### Authentication Policies

| Policy | What the user can do |
|--------|----------------------|
| **Password only** | The user signs in with their email and password. They cannot use SSO. This is the default for every user until you change it. |
| **Password or SSO** | The user can sign in either way — with their password or through the identity provider. |
| **SSO only** | The user must sign in through the identity provider. Password sign-in is refused. |

When a user set to **SSO only** tries to sign in with a password, they are told: **This account must sign in with Single Sign-On.**

<Note>
A user can belong to more than one company. Password sign-in is only fully blocked when **every** one of their company memberships mandates SSO. If any company they belong to still allows a password, they can sign in with their password and continue in that company.
</Note>

### The Linked SSO Provider

After a user signs in with SSO for the first time, their identity provider account is linked to their admin user. On the user's detail page, the **Authentication** section then shows a **Linked SSO provider** entry naming the provider.

To clear the link, click **Unlink SSO provider**. You are asked to confirm: **Are you sure you want to unlink the SSO provider for [name]? They will need to re-link the next time they sign in with SSO.** Unlinking is useful when a user needs to connect a different provider account, or when you want to revoke a stored link without changing the user's email address.

<Note>
You cannot unlink your own provider. This management action is only available to another user, which prevents you from locking yourself out where SSO is mandated.
</Note>

## The SSO Sign-In Experience

When SSO is enabled, the admin login screen shows one or more **Sign in with [provider]** buttons above the usual email and password fields, separated by an **Or continue with** divider.

To sign in with SSO:

1. Go to the admin login screen
2. Click **Sign in with [provider]**
3. Choose or confirm your account with the identity provider (with Google Workspace, the account chooser is always shown, even if you already have an active session)
4. You are returned to the admin area, signed in

If the platform cannot sign you in — for example your account is not enabled for SSO, your account is outside the configured workspace domain, or no matching admin user exists — you are returned to the login screen with: **We could not sign you in with SSO. Please check that your account has been enabled for this provider and try again.**

### Signing Out

If an **Admin logout URL** has been configured for your company, signing out of the admin area also sends you through the identity provider's logout flow before returning you to the login screen. If no logout URL is configured, you are simply returned to the admin login screen.

## Auditing

SSO activity is recorded so you can see how staff are signing in and review related changes:

- **Sign-in method is recorded.** Each sign-in attempt is tagged according to whether it used SSO or a password, so SSO sign-ins can be told apart from password sign-ins.
- **Sign-in problems are logged for diagnosis.** When an SSO sign-in fails — for example a domain mismatch, a missing claim, or an account that cannot be matched — the provider's response is recorded so the issue can be investigated.
- **User and policy changes are tracked.** Changes to a user, including their authentication policy and provider links, are captured in the platform's [audit logs](/core-platform/core/audit-logs) alongside all other administrative changes. You can also open a user's change history directly from their detail page using **View audit history**.

## Limitations

| Limitation | Detail |
|------------|--------|
| **Admin users only** | SSO covers staff signing in to the admin area. It does not change how customers sign in to the shop. |
| **Provider availability** | **Google Workspace** is the currently supported identity provider. The feature is designed to support additional providers in future. |
| **Staff must already exist** | SSO never creates new admin users. A person must already be added to your company as a user before they can sign in with SSO. |
| **MFA not enforced on SSO** | [Multi-factor authentication](/core-platform/system-access/mfa) is not additionally enforced when staff sign in through SSO — the identity provider is responsible for any additional verification. |
| **Configured by your account manager** | The provider connection is set up at the company level by your account manager. Within your admin area you control authentication on a per-user basis. |
