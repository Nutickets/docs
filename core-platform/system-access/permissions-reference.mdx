## Overview

This page lists every permission available when building a role in the admin area, organised by the area of the platform it applies to. Use it to answer two questions: *what does this permission actually let someone do?* and *what's the impact of granting it?*

It's a companion to [Roles & Permissions](/core-platform/system-access/roles-and-permissions), which explains how roles work and how to build one. Read that page first for the concepts; come here to look up an individual permission.

<Note>
The groups and permissions you see when building a role depend on which features are enabled for your company and on your own permissions — you can only grant abilities you hold yourself. If a group described here isn't visible to you, the relevant feature may not be enabled, or you may not have that ability.
</Note>

## How to Read This Reference

In the role builder, permissions are organised into **groups** by the type of data they apply to (Events, Orders, Venues, and so on). Within each group you choose two things: which **actions** to allow, and which **records** those actions apply to.

### Standard Actions

Most groups share the same standard set of actions. The label you see combines the action with the group name — for example **Edit** in the Events group reads as *"Edit Events"*.

| Action | What it allows |
|--------|----------------|
| **Manage** | A shorthand that grants **View**, **Edit**, **Delete**, and **Restore** for records in scope. It does **not** include Create or View list of. |
| **Create** | Create new records of this type. Must be granted explicitly. |
| **View list of** | See the full list/index of this record type. Must be granted explicitly. |
| **View** | Open and view an individual record. |
| **Edit** | Modify an individual record. |
| **Delete** | Delete a record (most records are soft-deleted and can be restored). |
| **Restore** | Restore a previously deleted record. |
| **Access** | A special action available on a few groups (such as Users) — see the relevant section below. |

<Warning>
**Manage is not "everything".** A user with **Manage Events** can view, edit, delete, and restore the events in their scope, but cannot create new events or open the full events list unless **Create** and **View list of** are also granted. Plan roles with this in mind.
</Warning>

### Which Records a Permission Applies To

For actions like View, Edit, Manage, and Delete, you also choose a visibility level that controls *which* records the permission covers:

| Visibility | Effect |
|------------|--------|
| **All** | Every record of that type. |
| **Selected** | Only the specific records you pick. |
| **Created by them** | Only records the user created themselves. |

See [Ability Visibility](/core-platform/system-access/roles-and-permissions#ability-visibility) for a fuller explanation, and [Visibility Groups](/core-platform/system-access/visibility-groups) for a separate, complementary way to limit what data a user can see.

<Note>
Many groups only offer the standard actions above. The sections that follow focus on **specialised permissions** — the ones that go beyond standard create/edit/delete and carry their own behaviour. Where a group is listed without specialised permissions, assume it offers the standard set.
</Note>

## Events & Ticketing

The **Events** group is the largest in the platform. Alongside the standard actions, it carries several specialised permissions that control event lifecycle and on-site selling.

| Permission | What it controls |
|------------|------------------|
| **Publish** | Take a draft event live. Without it, a user can build an event but not publish it — useful for an approval workflow where one team prepares events and another signs them off. See [Event Approvals](/core-platform/event-management/event-approvals). |
| **Cancel** | Cancel an entire event. See [Event Cancellations](/core-platform/event-management/event-cancellations). |
| **Edit sale data (items, seating, etc) for events** | Edit an event's sale structure — items, ticket types, products, seating plans, zones, timeslots, and similar — without necessarily being able to edit the event's core details. It is accepted in place of **Edit** wherever sale data is changed, so you can let someone build out tickets and seating while another role owns the event settings. |
| **If can't publish, can only edit stock after publishing** | A **restricting** permission. When a user holds it but cannot publish, then once an event is published they may only adjust stock/quantities — other event details become read-only. Use it to let staff top up availability on live events without changing anything else. |
| **Box office: sell items for** | Access box office mode and sell for the event. Covered in [Box Office and On-Site Sales](#box-office-and-on-site-sales). |
| **Email attendees** | Email the people booked onto an event. Covered in [Email Attendees](#email-attendees). |
| **Ignore box office validation: …** | Ten override permissions that let a box office operator bypass specific purchase rules. Covered in [Box Office Validation Overrides](#box-office-validation-overrides). |

<Note>
**Edit sale data vs Edit.** Pages that change an event's tickets, products, seating, or zones accept *either* **Edit** or **Edit sale data (items, seating, etc) for events**. Granting only the sale-data permission keeps a user out of the event's general settings while still letting them manage what's on sale.
</Note>

### Email Attendees

The **Email attendees** permission lets a user send emails to the people booked onto an event — for example to message ticket holders for a specific sale item or price band. It sits among the **Events** permissions when configuring a role, alongside **Publish** and **Cancel**.

Because it's an event permission, it follows the same visibility model as other event abilities: grant it with **All** to cover every event the user can access, **Selected** for specific events, or **Created by them** for events the user created. A user can only email attendees of events within the granted scope. See [Sending Emails](/core-platform/notifications/sending-emails).

<Note>
Email attendees is separate from editing an event. A user must be granted it explicitly to message attendees, even if they can already edit the event.
</Note>

## Orders, Refunds & Customers

The **Orders** group controls what a user can do with existing orders. Note it does not offer **Create** — orders are created through checkout, not the orders screen.

| Group | What it controls | Specialised permissions |
|-------|------------------|-------------------------|
| **Orders** | View and manage placed orders. | **Refund**, **Cancel**, **Apply cancellation fees to** — see below. |
| **Order items** | Individual line items within orders. | — |
| **Resale listings** | Tickets listed for [resale](/core-platform/resale/introduction). | **Cancel** — cancel a resale listing. |
| **Customers** | [Customer](/core-platform/customer-management/managing-customers) records. | — |
| **Wallets** | Customer [account balances](/core-platform/account-balances/top-ups). | **Cashout** — pay a customer's wallet balance back out. |
| **Wallet transactions** | Individual balance transactions. | **Cancel** — cancel a wallet transaction. |

<Note>
**Refund vs Cancel.** **Refund** returns money against an order; **Cancel** cancels the order (and can apply cancellation fees). They're separate permissions — a role can be allowed to refund but not cancel, or vice versa. See [Refunds & Cancellations](/core-platform/order-management/refunds-and-cancellations).
</Note>

<Warning>
**Apply cancellation fees to** changes the *money* a customer gets back. If a user cancels an order **without** this permission, any cancellation fee configured on the items is forced to zero — the customer is refunded in full. Only grant it to staff who should be able to retain cancellation fees on a customer's behalf.
</Warning>

<Warning>
**Wallets → Cashout** moves real money out of a customer's balance. Treat it as a financial permission and restrict it to trusted staff. See [Cash-outs](/core-platform/account-balances/cash-outs).
</Warning>

## Box Office and On-Site Sales

These permissions govern selling in person through [box office mode](/core-platform/box-office/introduction) and the [EPOS](/core-platform/epos/overview) point of sale.

### Selling for an Event

The **Box office: sell items for** permission (in the **Events** group) controls whether a user can enter box office mode and sell for an event.

<Note>
When granted with **All**, this permission effectively means *"can access box office mode"* rather than *"can sell for literally every event"* — it's the switch that turns box office on for the user. Users can always sell for events they created themselves. Visibility (**Selected** / **Created by them**) then narrows which events they can sell for.
</Note>

### Box Office Validation Overrides

The checkout normally enforces the same purchase rules at the box office as online — stock limits, on-sale dates, quantity rules, and so on. The ten **Ignore box office validation** permissions (in the **Events** group) let a trusted operator override a specific rule when selling in person.

| Permission | What it lets the operator do |
|------------|------------------------------|
| **Ignore box office validation: Oversell** | Sell beyond remaining stock and event, ticket-type, and item capacity limits. |
| **Ignore box office validation: Data capture** | Complete a sale without answering mandatory data capture questions. |
| **Ignore box office validation: On-sale dates** | Sell items that are outside their on-sale window. |
| **Ignore box office validation: Min & max order quantity** | Sell quantities above or below the per-order minimum/maximum. |
| **Ignore box office validation: Forced minimum order quantity** | Skip a forced minimum order quantity. |
| **Ignore box office validation: Must purchase in multiples of** | Sell quantities that aren't the required multiple. |
| **Ignore box office validation: Limit item purchases (per customer)** | Exceed per-customer purchase limits and single-purchase-only restrictions. |
| **Ignore box office validation: Must be purchased with other items** | Sell items that normally require accompanying items. |
| **Ignore box office validation: Must book entire table, not individual seats** | Sell individual seats from a table that must otherwise be booked whole. |
| **Ignore box office validation: Min/Max event team size** | Book [group/team events](/core-platform/event-management/group-team-events) outside their team-size limits. |

<Warning>
These overrides let staff bypass rules that protect stock, capacity, and data quality. Grant them only to senior box office staff, and scope them to the events that need them.
</Warning>

### Box Office Capabilities & Payment Methods

The **Box office** group controls what tools an operator can reach during a sale, independent of any one event.

| Permission | What it controls |
|------------|------------------|
| **Box office: Can apply external vouchers** | Apply [external vouchers](/core-platform/basket-codes/external-vouchers) during a box office sale. |
| **Box office: Can apply custom discounts** | Create and apply ad-hoc custom discounts during a box office sale. See [Discounts & Overrides](/core-platform/box-office/discounts-and-overrides). |
| **A permission per payment method** | One toggle for each box office [payment method](/core-platform/box-office/payment-methods) — for example Cash, Card present, Card not present, Bank transfer, Complimentary, Invoice, Gift card, Cheque, Money order, Payment link, Reserve to pay onsite, and Split payment. Enabling a method's permission lets the operator take payment that way; leaving it off hides it at checkout. |

<Note>
Payment-method permissions are enforced at the point of sale: only the methods an operator has been granted appear as options when taking payment. This lets you, for example, allow a cashier to take Cash and Card but not write off sales as Complimentary.
</Note>

### EPOS

The **EPOS shops** group governs the [EPOS](/core-platform/epos/overview) point of sale, with three specialised permissions on top of the standard actions:

| Permission | What it controls |
|------------|------------------|
| **Box office: sell items for** | Sell products at the EPOS till. |
| **Refund** | Refund EPOS transactions. |
| **Amend price** | Manually adjust an item's price at the till. |

The **EPOS products** group controls who can create and manage the products sold through EPOS.

<Warning>
**Users → Access** lets a user sign in *as* another user to see exactly what they see. It's invaluable for troubleshooting, but it grants that user the other account's view of the system — restrict it to administrators and support staff. See [Accessing a User's Account](/core-platform/system-access/users#accessing-a-users-account).
</Warning>

<Note>
**Manage roles is a powerful permission.** Because a user can grant any ability they hold themselves, anyone who can manage roles can extend their own reach through the roles they build. The platform prevents a user from granting abilities they don't already have, but a broadly-permissioned user with role management can still create powerful roles. Grant it deliberately.
</Note>

<Note>
**Dev tools** are reserved for Super Users and are never offered as a normal role permission.
</Note>

## The Super User Role

The **Super user** role bypasses every permission check on this page — a super user can do anything in the admin area, including actions that can't normally be granted. Assigning it shows a confirmation warning. Use it only for users who genuinely need unrestricted access. See [Super User](/core-platform/system-access/roles-and-permissions#super-user).
